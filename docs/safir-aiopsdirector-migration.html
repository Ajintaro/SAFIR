1 Was ist SAFIR?

Sprachgestützte Assistenz für Informationserfassung in der Rettungskette. Ein KI-gestütztes Dokumentationssystem entlang der Rettungskette der Bundeswehr (Sanitätsdienst).

Kernprinzip: Verwundete müssen innerhalb der Goldenen Stunde (60 Min.) medizinisch versorgt werden. SAFIR digitalisiert die Patientenerfassung per Sprache und leitet die Daten automatisch durch die Rettungskette weiter.

Role 3Erweiterte VersorgungEinsatzlazarettPatientenakte, Diagnose-KI
Role 4RehabilitationBW-KrankenhausStatistik, Auswertung

8 Migrations-Workstreams

WS1 Containerisierung

Jede Komponente bekommt ein eigenes Dockerfile:

  • safir-api — Python 3.12, FastAPI, uvicorn, httpx, python-docx
  • safir-whisper — CUDA Base Image + whisper.cpp (oder faster-whisper)
  • safir-llm — Ollama Container (offizielles Image: ollama/ollama)
  • safir-diarize — PyTorch + pyannote-audio
  • safir-dashboard — Nginx + statische HTML/JS/CSS Dateien
  • safir-sitaware — Python + CoT/NVG Generator
WS2 Kubernetes Deployment
  • Helm Charts oder AIOps Director Devenor Templates für jedes Microservice
  • GPU Node Pool (Azure NC-Series oder ND-Series) für Whisper, LLM, Diarization
  • nvidia.com/gpu: 1 Resource Requests für GPU-Pods
  • Horizontal Pod Autoscaler für safir-api
  • Persistent Volume Claims für Audio-Dateien und Protokolle
WS3 Storage Migration
  • JSON-Dateien → MongoDB (Patienten, Events, Timeline) — dokumentenorientiert, passt ideal zum verschachtelten PATIENT_SCHEMA
  • WebSocket State → Redis PubSub (für Multi-Replica API)
  • Audio-Dateien + DOCX → Azure Blob Storage
  • Migrationsscript für bestehende JSON-Daten → MongoDB Collections
WS4 Netzwerk & VPN
  • Tailscale ersetzen durch Azure VPN Gateway (IPsec/WireGuard)
  • Jetson Feldgeräte verbinden sich per VPN-Client zum Azure VNET
  • Ingress Controller (Nginx/Traefik) für HTTPS-Zugriff auf Dashboards
  • mTLS zwischen Services (Kubernetes Network Policies)
  • Jetson BACKEND_URL zeigt auf Azure-Ingress statt auf lokale IP
WS5 AIOps Director Integration
  • Registrierung der SAFIR Microservices in der Service Registry
  • Integration mit AIOps Director Message Queue für async Processing
  • AIOps Director Monitoring (Prometheus/Grafana) für GPU-Auslastung
  • AIOps Director Admin Microservice für Konfiguration
  • SDK-Template (Devenor) für SAFIR Microservice-Basis
  • Optional: SAFIR als "Customer AI Orchestration" in AIOps Director einbinden

9 Offene Fragen an das AIOps-Team

❓ Klärungsbedarf
  1. GPU-Verfügbarkeit: Welche GPU-Instanzen stehen im Azure-Cluster zur Verfügung? (Min. 24 GB VRAM für alle Modelle, idealerweise A100/A10G)
  2. Ollama vs. vLLM: Wird Ollama auf Kubernetes unterstützt, oder soll auf vLLM / TGI (Text Generation Inference) umgestellt werden?
  3. Devenor SDK: Können wir das Microservice-Template für SAFIR nutzen? Wie sieht das Basis-Template aus?
  4. Service Registry: Wie registrieren sich neue Microservices? REST? gRPC? Auto-Discovery?
  5. Message Queue: Welches System? (RabbitMQ, Kafka, NATS?) — Für async Audio-Processing wichtig
  6. Ingress & Auth: Wie werden externe Clients (Jetson im Feld) authentifiziert? mTLS? API-Keys? OAuth?
  7. Air-Gapped Support: SAFIR muss in Einsatzgebieten ohne Internet funktionieren. Unterstützt AIOps Director On-Prem / Air-Gapped Deployment?
  8. WebSocket Support: Unterstützt der Ingress Controller WebSocket-Verbindungen (für Live-Transkription)?
  9. Storage: Gibt es einen managed MongoDB (z.B. CosmosDB mit Mongo-API) und Redis im Cluster, oder müssen wir eigene deployen?
  10. CI/CD Pipeline: Wie werden neue Container-Images gebaut und deployed? GitLab CI? Azure DevOps?
  11. NATO-Compliance: Gibt es Anforderungen an Verschlüsselung / Zertifizierung (VS-NfD, GEHEIM)?
  12. CLARA Integration: SAFIR's Speech-to-Text und CLARA (AIOps Director) haben Überschneidungen. Sollen wir CLARA als Whisper-Ersatz evaluieren?
